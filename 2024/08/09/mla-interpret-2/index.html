MHA（2）： Multi-head Latent Attention (MLA) 在说什么

Introduction

在上一节中, 我们介绍了Multi-head Attention以及接下来的变种MQA以及GQA, 那么这一篇我们就主要来讲讲之后的新创新 Multi-head Latent Attention (MLA)

回顾GQA

在进入正题之前, 我需要先纠正一下上一篇中的一些内容.

在上一篇中, 我把GQA的过程描述成从num_heads中选择num_key_value_heads个头来作为key, 这种说法是不严谨的. 事实上, 从构建模型的时候, 我们就限制key与value的头数, 也就是, 限制模型参数的 WkW_kWk​ 与 WvW_vWv​ 这两个矩阵的大小.

如果按照一般的MHA, 那么k_proj与v_proj都应当是一个方阵, 长度为hidden_size, 但是在GQA中, k_proj与v_proj本身就不再是方阵, 这点我们可以找一个使用GQA的模型代码来看, 下面是Mistral这个模型的transformers库中的代码细节

从上面的代码可以看到, WkW_kWk​ 与 WvW_vWv​ 本身其实就已经是一个长方形的矩阵了, 短边从hidden_size缩减到num_key_value_heads * head_dim, 所以, 在模型构建的层面, GQA本身就已经决定了他不能再上采样回到MHA的精度, 但是可以通过重复每一个head, 扩展回到MHA的维度, 也就是使用repeat_interleave这样的操作来实现注意力头这一维度的扩展.

在谈及MLA的原理之前, 笔者还是想要先从代码出发, 来看看MLA到底对输入的Tensor, 大小为 (batch_size, seq_length, hidden_size), 做了一些怎样的操作.

attention中的前传流程

这里可以看到, 相比较于一般的llama形式的attention操作, MLA版本的attention层有以下几点特殊

  • q先经过降维, 又重新升维, 并进行了拆分
  • k与v都是通过compressed_kv进行升维得到的
  • RoPE编码并不是作用在q或者k变量上, 而是作用在别的tensor上, 最后与q和k进行拼接, 并进入attention计算

画出了整体的attention计算流程, 图中的B表示Batch_Size, S表示Sequence_Length, 蓝色部分代表中间变量, 绿色部分代表模型参数.

整体流程可以分成两步

  • 通过输入的隐层hidden_state, 也就是图中的X, 计算q, k, v, 其中q与k分成两个部分, 分别是(q_nope, k_nope)与(q_rope, k_nope)
    • q(k)_nope代表不进行RoPE计算的Tensor
    • q(k)_rope代表进行RoPE计算的Tensor

  • q(k)_rope经过RoPE计算之后, 与原本的q(k)_nope拼接, 生成query_state与key_state, 并与v_nope进行attention计算, 这里的attention计算就是非常标准的MHA. 最后与o_proj生成该层的输出

图中的B表示Batch_Size, S表示Sequence_Length

图中的蓝色部分代表中间变量, 绿色部分代表模型参数

整体的流程其实很清楚了, 那么MLA的优化点, 就在于图中的橙色部分, 也就是那个compressed_kv, 显然, 在推理的时候, 我们可以直接使用compressed_kv, 来重构k与v, 相比较于直接存储k和v而言, 显然存储一个低维的compressed_kv, 能很大程度上的节约KVCache, 这也就是MLA的最大优势.

显存的计算

这一节中我们来具体计算一下MLA所消耗的显存, 以及对比之下, MLA比起MHA与GQA, 能够节约多少显存.

为了有一个同尺度的比较, 我们假设有这样一个模型, 它有着与Deepseekv2一样的参数设置, 但是使用的是普通的llama结构, 也就是说, 我们初始化了一个llama模型, 它的attention层的参数是

  • hidden_size = 5120
  • num_heads = 128
  • head_dim = 128
  • 如果使用了GQA, 那么就假设num_key_value_heads = 128 // 8 = 16

出于简单的目的, 下面计算的kvcache均指一层Layer中一个batch中的一个token的参数量

下面讨论的 K = 1024, M = 1024 x 1024

如果模型使用的是MHA

模型本身的一个attention block的参数量为

  1. q_proj: (5120, 5120)
  2. k_proj: (5120, 5120)
  3. v_proj: (5120, 5120)
  4. o_proj: (5120, 5120)

可以算出总参数量为5120 x 5120 x 4 = 100M

同样, 我们也可以得到, kvcache的参数量为5120 x 2 = 10K, 如果使用bf16格式保存, 则需要20KByte的空间

如果模型使用的是GQA

模型的一个attention block参数量为

  1. q_proj: (5120, 5120)
  2. k_proj: (5120, 2048)
  3. v_proj: (5120, 2048)
  4. o_proj: (5120, 5120)

可以算出总的参数量为 70M

kvcache的参数量为2048 x 2 = 4K, 使用bf16格式保存需要8KByte的空间

如果模型使用的是MLA

当然这里MLA的参数设置其实有任意性, 我们只谈论与Deepseekv2相同的模型结构

一个attentionblock的参数量为

  1. q_a_proj: (5120, 1536)
  2. q_b_proj: (1536, 24576)
  3. kv_a_proj_with_mqa: (5120, 576)
  4. kv_b_proj: (5120, 32768)
  5. o_proj: (16384, 5120)

总的参数量为 300,220,416 = 286.3125M

kvcache的参数量为576 = 0.5625K, 使用bf16格式保存需要1.125KByte的空间

总结

根据上面的计算, 我们可以列出下面的表格

Attention方式 模型参数量 KVCache参数量
MHA 100M 10K
GQA 70M 4K
MLA 286.3M 0.56K

从上面的数据就可以得到我们的结论

MLA就是在扩大模型参数量的同时, 大大降低了KVCache的参数量

一般情况下, 如果没有多实例, 模型的参数只会保存一份, 但是KVCache的数量会随着输入的增加逐渐增加. 因此, 这样的对模型结构的修改如果不影响推理精度, 那在占用显存空间方面就是值得的.

但是, 额外增加的参数量, 也确实增加了推理时的计算量, 因此, 这就是根据具体情况的取舍了.

隐藏在MLA之后的数学

其实这一部分笔者并不擅长, 感兴趣的读者可以阅读苏神的这篇博客. 如果上面的内容能够理解, 相信数学部分的推导也不是难事.

但是有一个有趣的现象还是想单拿出来说一说

一般我们认为, QQQ, KKK, VVV 应当是对于原始文本的三种表征, 这样才会使得Attention本身是合理的, 因为我们希望经过训练, QQQ, KKK, VVV 本身应当能够分别代表Query, Key, Value三种不同的含义.

但是在计算中, 我们的 QQQ, KKK, VVV 的计算遵从的是下面的公式

Q=WQXK=WKXV=WVXQ = W_Q X \\ K = W_K X \\ V = W_V X \\ Q=WQ​XK=WK​XV=WV​X

那么, 很简单的道理, 如果 WKW_KWK​ 是方阵且恰非奇异, 那么我们有

V=WVWK−1KV = W_V W_K^{-1} K V=WV​WK−1​K

也就是说, VVV 其实就是 KKK 经过一个线性变换得到的向量. 换言之, 如果三个WWW都非奇异, 那么 QQQ, KKK, VVV 都是彼此的线性变换. 即使矩阵是不满秩的, 我们也可以计算近似. 这样, 我们的结论就是 VVV 总可以由 KKK 去经过一个线性变换近似或者精确表达.

那么MLA的思路就非常的顺理成章了, 既然 VVV 到 KKK 是一个线性变换, 那么我用一个独立的向量同时表征 VVV 和 KKK 也是合情合理的. 至于这个向量长度多少, 那就是损失多少信息的事情了, 问题反而不是很大.

Conclusion

本文主要探讨了Multihead Latent Attention (MLA)的具体实现以及一些想法. 事实上, 受限于能力原因, 笔者一开始看了很多MLA相关的数学推导, 但是一直无法理解, 后来才想到从代码入手, 看看具体是如何实现的, 然后倒推数学, 这才有了一些感想. 所以本文也是从代码流程入手, 数学方面就不献丑了.

另外, MLA这个算法与其说是对KVCache的修改, 不如说是对模型结构的一次大改, 原本的GQA结构姑且还可以说是MHA的小小修改, 而MLA已经将MHA的流程改的面目全非. 这也让原本基于MHA的模型, 几乎无法修改成MLA的结构. 这大概就是为什么MLA出来了这么久, 而且KVCache的精简效果足够好, 却一直不温不火吧.
